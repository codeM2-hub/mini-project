## 🛠️ 2. Preprocessing & Automatic Dataset Generation
**Script:** `scripts/build_speaker_dataset.py`

This script handles the heavy lifting of turning raw audio into a robust ML dataset:
1. **Silence Removal:** It scans the 7-minute files in `anomaly_raw/` using Librosa's `effects.split()` to skip silent periods (so the model learns *voices*, not silence).
2. **Windowing:** It slices the remaining active audio into overlapping **3.0-second** chunks.
3. **Aggressive Augmentations:** To make the model robust against real-world background noise (mobile mics, fans, distance), it applies extreme augmentations using the `audiomentations` library:
   - **Gaussian Noise:** Simulates cheap microphones or fan noise.
   - **Time Stretch:** Simulates people speaking faster or slower.
   - **Pitch Shift:** Slightly alters pitch to broaden the "anomaly" spectrum.

*(Note: It generated over **3,500 Valid** samples and **1,700 Anomaly** samples during the recent run!)*

## 🧠 3. Speaker Verification CNN Model
**File:** `ml/models/speaker_cnn.py`

Instead of the previous multi-class CRNN, I built a highly robust Convolutional Neural Network specifically for binary classification (Valid vs. Anomaly).
* **Deep Feature Extraction:** Uses 4 blocks of 2D Convolutions (`Conv2d -> BatchNorm2d -> MaxPool2d -> Dropout2d`).
* **Batch Normalization:** Stabilizes the learning process and reduces internal covariate shift.
* **Aggressive Dropout:** We use Dropout layers up to `0.5` before the fully connected layer. This acts as extreme regularization to prevent the model from memorizing your voice and forces it to learn underlying vocal tract characteristics.
* **Global Average Pooling:** Allows the network to handle slightly varying audio lengths seamlessly.

## 📈 4. Training Pipeline & Class Imbalance
**Script:** `scripts/train_speaker.py`

1. **Feature Extraction:** It automatically converts the thousands of WAV files into Log-Mel Spectrograms `(Batch, 1, 64, Time)` and caches them to disk (`data/speaker_features.npz`) so future runs are lightning-fast.
2. **Class Imbalance Handling:** Because you might have more anomaly clips than valid clips (or vice versa), the script calculates specific `weight` tensors and passes them into the `CrossEntropyLoss` function.
3. **Early Stopping & Checkpointing:** It monitors the **F1-Score** on the validation set. If the F1-Score stops improving for 7 epochs, it stops training to prevent overfitting.

## 🔐 5. Inference Engine & Aggressive Threshold Logic
**File:** `ml/inference/speaker_verifier.py`

This is where the magic happens for reducing **False Positives** (which was your main issue). 

Standard models classify anything > `0.5` as True. In speaker authentication, `0.5` is extremely dangerous because a TV voice might sound *slightly* like a human, getting a `0.6` confidence.

**The Solution:**
```python
CONFIDENCE_THRESHOLD = 0.92

prob_valid = float(probs[1])

if prob_valid > CONFIDENCE_THRESHOLD:
    return "✅ Authorized Voice Detected"
elif prob_valid > 0.5:
    return "🚨 Access Denied (Voice similar but failed strict threshold)"
else:
    return "🚨 Access Denied (Anomaly detected)"
```
By forcing the model to be **> 92% confident**, it will ruthlessly reject YouTube videos, TV audio, and random people speaking.

## 🚀 Best Practices for Speaker Verification
To ensure this system stays bulletproof:
1. **Enrollment Quality:** The quality of your `voice_data` (your voice) dictates the model's accuracy. Record yourself in different rooms, at different distances from the mic, and at different times of day (morning voice vs. evening voice).
2. **Negative Mining:** If the system ever incorrectly accepts a YouTube video, record 10 seconds of that exact video, place it in `anomaly_raw/`, and re-run the `build_speaker_dataset.py` -> `train_speaker.py` pipeline. The model will learn from its exact mistake.
3. **Silence Checks:** The inference engine now computes the RMS (Root Mean Square) energy of the incoming audio window. If it's pure silence, it immediately returns `ANOMALY` without even waking up the CNN, saving CPU and preventing bizarre predictions on static noise.
